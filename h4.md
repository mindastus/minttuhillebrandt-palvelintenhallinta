## h4 Pizza Fantasia

## x) 4.12.1 Size and Complexity of Some DSLs
- Salt DSL: 510 state-funktiota, dokumentaatio yli 20 000 riviä / 75 000 sanaa – isompi kuin väitöskirja.
- Lisäksi käytetään Jinja2-templaatteja, jotka lisäävät uuden abstraktiotason ja monimutkaisuutta.
- Puppet: 113 funktiota, mutta omat, epätavalliset kontrollirakenteet (esim. virtual resources).
- Johtopäätös: Suosittujen CM-työkalujen DSL:t ovat suuria, monimutkaisia ja vaikeita hallita.
- Oma huomio: Onko näin laaja DSL oikeasti hyödyllinen, jos suurin osa käyttäjistä tarvitsee vain murto-osan toiminnoista?

## x) 4.12.2 Use of DSL Functions in Case Configuration
- Todellisissa Puppet-konfiguraatioissa pieni joukko funktioita kattaa suurimman osan käytöstä.
- Mozilla Release Engineering: 87 % käytöstä kattoi vain muutama komento:
file, package, service, exec, case, if jne.
- USGCB: yleisimmät olivat file, service, exec, package, augeas.
- Käyttäjät eivät käytä suurta osaa DSL:n tarjoamista ominaisuuksista.
- Tyypillinen kuvio: package → file → service (daemonien asennus).
- Oma kysymys: Jos 80–90 % käytöstä on samoja peruspalikoita, pitäisikö CM-työkalujen DSL:n olla paljon pienempi ja selkeämpi?

## x) 4.12.3.1 Dependencies Between Main Functions
- Kaikki tärkeät resurssit voidaan rakentaa kahden perusasian päälle:
file (file, directory, symlink)
exec (komentojen suoritus)

- Näistä muodostuvat:
package, service, user, group

- Idempotenssi saavutetaan yksinkertaisilla if-rakenteilla (tarkista → korjaa → raportoi).
- Useimmat CM-toiminnot ovat lopulta vain komentojen suorittamista tai tiedostojen hallintaa.
- Korkeamman tason resurssit (esim. cron) voidaan rakentaa näiden päälle.
- Oma huomio: On yllättävää, että koko CM-järjestelmän voi pohjimmiltaan rakentaa vain kahden perusresurssin varaan – pitäisikö kaikkien työkalujen olla näin yksinkertaisia?

Ja sitten ajaa ansible-roolisi uudestaan ja todeta, että se korjaa tilanteen.
