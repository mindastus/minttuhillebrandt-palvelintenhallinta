## h4 Pizza Fantasia

## x) 4.12.1 Size and Complexity of Some DSLs
- *Salt DSL:* 510 state-funktiota, dokumentaatio yli 20 000 riviä / 75 000 sanaa – isompi kuin väitöskirja.
- *Puppet:* 113 funktiota, mutta omat kontrollirakenteet (esim. virtual resources).
- *Yhteenveto:* Suuret CM‑DSL:t ovat raskaita ja vaikeita hallita.
- *Oma huomio:* Tarvitaanko oikeasti näin isoja DSL:iä, jos käyttäjät käyttävät vain murto-osaa?

## x) 4.12.2 Use of DSL Functions in Case Configuration
- Todellisissa Puppet‑konfiguraatioissa 80–90 % käytöstä on samoja perusjuttuja:
`file, package, service, exec, if/case.`
- *USGCB:* samat perusfunktiot + augeas.
- Suuri osa DSL:n ominaisuuksista jää käyttämättä.
- *Oma kysymys:* Jos käytetään vain muutamaa perusfunktiota, pitäisikö työkalujen olla paljon yksinkertaisempia?

## x) 4.12.3.1 Dependencies Between Main Functions
- Kaikki rakentuu kahden perusasian päälle: `file` ja `exec`.
- Näistä muodostuvat: `package`, `service`, `user`, `group`.
- Idempotenssi = tarkista → tee muutos vain jos tarvitaan.
- Korkeamman tason resurssit (esim. cron) voidaan rakentaa näiden päälle.
- *Oma huomio*: Koko CM-järjestelmä voidaan pohjimmiltaan rakentaa vain kahdesta peruspalikasta.

2. Yritys 

Kokeilin seuraavaa:

```
sudo rm -r /etc/nginx/
```

- **Odotus:** Nginx ei käynnisty → Ansible luo hakemiston ja asetukset uudelleen.  
- **Todellisuus:** Palvelin ei mennyt kunnolla rikki, tai Ansible ei reagoinut hakemiston puuttumiseen. Rooli ei tehnyt muutoksia, eikä tilanne muuttunut.
- **Johtopäätös:** Tämäkään ei aiheuttanut sellaista virhettä, jonka rooli olisi korjannut.

# Yhteenveto

> Kokeilin useita tapoja rikkoa palvelin. Ensimmäinen yritys ei aiheuttanut odotettua virhettä, koska järjestelmä ei mennyt kunnolla rikki ja Ansible ei tehnyt muutoksia.  
> Toisessa yrityksessä poistin koko `/etc/nginx/`‑hakemiston, mutta myöskään tämä ei johtanut tilanteeseen, jossa rooli olisi korjannut palvelun.  
> Lisäksi opin, ettei tekoälyyn kannata luottaa sokeasti, kaikki ehdotetut tavat eivät toimineet käytännössä.

## e) Idempotentti. Osoita, että tilasi on idempotentti.

- Idempotenssi tarkoittaa, että kun sama Ansible‑playbook ajetaan useita kertoja peräkkäin, ensimmäinen ajo tekee muutokset, mutta toinen ja kolmas ajo eivät muuta mitään, koska palvelin on jo halutussa tilassa. Tämä näkyy siten, että tehtävien tila muuttuu changed → ok. Koska en saanut edellistä tehtävää tehtyä loppuun, niin tässä alla on teoreettinen esimerkki siitä, miltä idempotentti ajo näyttäisi.

Ensimmäinen ajo, muutoksia tapahtuisi
- `ansible-playbook site.yml --ask-become-password`
```
TASK [Install nginx]
changed: [server1]

TASK [Copy index.html]
changed: [server1]

TASK [Ensure nginx is running]
changed: [server1]

PLAY RECAP
server1 : ok=0  changed=3  failed=0
```
- Ensimmäinen ajo asentaa Nginxin, kopioi tiedoston ja käynnistää palvelun.

Toinen ajo – ei muutoksia (idempotentti)
- `ansible-playbook site.yml --ask-become-password`
```
TASK [Install nginx]
ok: [server1]

TASK [Copy index.html]
ok: [server1]

TASK [Ensure nginx is running]
ok: [server1]

PLAY RECAP
server1 : ok=3  changed=0  failed=0
```
- Toisella ajolla kaikki tehtävät ovat ok, eikä yhtään changed‑tilaa näy.
Rooli ei tee turhia muutoksia → idempotentti tila saavutettu.

- Ansible‑rooli on idempotentti, koska kun ajan saman playbookin uudelleen ilman muutoksia, kaikki tehtävät palauttavat tilan ok eikä yhtään changed‑tilaa ilmesty. Tämä osoittaa, että palvelin on jo halutussa tilassa eikä Ansible tee mitään ylimääräistä.
