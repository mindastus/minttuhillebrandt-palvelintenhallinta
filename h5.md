## h5 Gitar Hero

## x) Chacon and Straub 2014: Pro Git, 2ed: 1.3 Getting Started - What is Git?
- Lähes kaikki Git‑operaatiot ovat paikallisia.
- Git varmistaa datan eheyden.
- Gitin tietokanta kasvaa lisäämällä uusia objekteja, ei poistamalla vanhoja.
- **Oma huomio:** Git on kuin pieni tiedostojärjestelmä, jonka päällä on tehokkaita työkaluja.

## x) git add --all | Git: git-add documentation | https://git-scm.com/docs/git-add
- `git add --all` lisää uudet tiedostot sekä päivittää muutetut.
-  `git add` voidaan ajaa useita kertoja ennen commitia.
- Komento ei lisää oletuksena ignored‑tiedostoja, ellei käytetä `--force`.
- **Oma huomio:** `--all` / `-A` toimii samalla tavalla, päivittää indeksin vastaamaan koko työpuuta.

## x) git commit | Git: git-commit documentation | https://git-scm.com/docs/git-commit
- `git commit` luo uuden commitin staging‑alueen sisällöstä ja viestistä.
- Commitin sisältö tulee:
  ```
  – staging‑alueelta (git add)
  – automaattisesti -a‑valinnalla
  – interaktiivisesti (--patch, --interactive)
  – tai suoraan tiedostopolkujen perusteella
  ```
- **Oma huomio:** `commit` voi sisältää myös allekirjoituksen

## x) git pull | - Git: git-pull documentation | https://git-scm.com/docs/git-pull
- `git pull` hakee muutokset ja yhdistä ne
- Oletuksena tekee `fast‑forward`‑päivityksen, jos mahdollista
- Jos tulee konflikteja, ne voi perua.
- **Oma huomio:** Helppo käyttää, mutta taustalla tapahtuu paljon.

## x) git push | Git: git-push documentation | https://git-scm.com/docs/git-push
- `git push` lähettää paikalliset commitit etärepoon ja päivittää siellä olevat viitteet.
- Lähettää vain commitit ja objektit, joita etärepossa ei vielä ole.
- **Oma huomio:** Voit vahingossa pyyhkiä muiden tekemät commitit.

## a) Online. Tee uusi varasto GitHubiin (tai Gitlabiin tai mihin vain vastaavaan palveluun). Varaston nimessä ja lyhyessä kuvauksessa tulee olla sana "sunshine". Aiemmin tehty varasto ei kelpaa. (Muista tehdä varastoon tiedostoja luomisvaiheessa, esim README.md ja GNU General Public License 3) Edistyneemmille voi olla hauskaa etsiä ja kokeilla jokin muu palvelu kuin Github.

## b) Dolly. Kloonaa edellisessä kohdassa tehty uusi varasto itsellesi, tee muutoksia omalla koneella, puske ne palvelimelle, ja näytä, että ne ilmestyvät weppiliittymään.

- Kloonaa varasto omalle koneelle
`git clone https://github.com/kayttaja/repo.git`

<img width="425" height="198" alt="image" src="https://github.com/user-attachments/assets/f21bc51b-ddd0-4d9f-8c91-8d3ab8776e15" />

- Tee muutoksia paikallisesti
`nano uusi.txt`

<img width="319" height="80" alt="image" src="https://github.com/user-attachments/assets/994c9f54-cb47-4463-a815-572a31bd74f8" />

- uusi.txt sisältö

<img width="355" height="182" alt="image" src="https://github.com/user-attachments/assets/aace13b1-0129-4c4f-8b42-981b9acb9843" />

- Lisää kaikki muutokset staging‑alueelle:
`git add --all`

- Luo commitin staging‑alueen sisällöstä:
`git commit` > `Add one .txt for test`

- Hakee etäreposta mahdolliset uudet commitit ja yhdistää ne. Tuloste “Already up to date.” kertoo, ettei etärepossa ollut mitään uutta:
`git pull`

- Lähettää paikalliset commitit GitHubiin:
`git push`

<img width="454" height="193" alt="image" src="https://github.com/user-attachments/assets/f7d46190-7efa-461f-a42e-88dd951556d2" />

- Katsotaan että virtuaalikoneessa tehdyt muutokset Githubiin näkyy myös selaimessa.

<img width="331" height="200" alt="image" src="https://github.com/user-attachments/assets/36b03056-2455-411e-a94c-d3c6d76ff571" />

- Tarkistetaan että uusi.txt sisältö sama kuin aikaisemmin.

- Kloonasin aiemmin luomani sunshine‑varaston omalle koneelleni komennolla `git clone`. Tämän jälkeen tein paikallisesti muutoksia ja tallensin ne: `git add --all`, `git commit` ja `git push`. Kun päivitys oli työnnetty GitHubiin, tarkistin selaimessa, että muutokset näkyivät varaston weppiliittymässä, mikä vahvisti onnistuneen pushin.

## c) Doh! Tee tyhmä muutos gittiin, älä tee commit:tia. Tuhoa huonot muutokset ‘git reset --hard’. Huomaa, että tässä toiminnossa ei ole peruutusnappia.

<img width="376" height="89" alt="image" src="https://github.com/user-attachments/assets/903fa681-ea99-473a-b3f6-8dda051365f5" />

- Tee “tyhmä muutos”. Lisäsin vain yhden rivin tkestiä lisää.

<img width="383" height="196" alt="image" src="https://github.com/user-attachments/assets/a67dbc76-6cc9-4568-be4f-44bcda828714" />

- Tarkista että Git huomaa muutoksen:
`git status`

- Tuhoa muutokset:
`git reset --hard` > Palauttaa tiedostot viimeisimpään commitiin.

<img width="322" height="81" alt="image" src="https://github.com/user-attachments/assets/4de54068-55e4-4cda-97cc-ea91747426a6" />

- Tarkista että muutos katosi:
`git status`

- Avasin myös uusi.txt uudestaan, että näin että muutos oikeasti katosi.

- Tein tarkoituksella huonon muutoksen paikalliseen Git‑varastoon, mutta en tehnyt commitia, jotta muutos jäi pelkästään working treeen. Kun halusin perua koko sotkun, käytin komentoa `git reset --hard`, joka palautti projektin viimeisimmän commitin tilaan ja poisti kaikki tekemäni tallentamattomat muutokset. Tämä komento on lopullinen eikä sille ole peruutusnappia, joten sitä kannattaa käyttää vain silloin, kun varmasti haluaa hävittää kaikki paikalliset muutokset.
